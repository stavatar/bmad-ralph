## Найденные проблемы и решения

---

### CRITICAL

#### BUG-1: Счётчик завершённых задач всегда показывает 0

Ralph завершил 5 задач успешно, но в итоге написал `0 completed`. Задачи финишируются со статусом `"done"`, а счётчик ищет статус `"completed"`. Одно слово не совпадает — и вся статистика врёт.

- **Файл:** `runner/metrics.go:244` — `case "completed":` в `Finish()`
- **Вызов:** `runner/runner.go:1320` — `r.Metrics.FinishTask("done", headSHA)`
- **Результат:** `tasks_completed: 0` в JSON-отчёте при 5 реально завершённых задачах
- **Решение:** Передавать `"completed"` в FinishTask везде (runner.go + все тесты)

#### BUG-2: Ralph останавливается целиком когда одна задача не прошла ревью

После 5 неудачных ревью-циклов на задаче BF-2.1 Ralph не перешёл к следующей задаче, а прекратил весь запуск. Из-за этого 7 задач из 13 вообще не были начаты.

- **Файл:** `runner/runner.go:1200` — `return fmt.Errorf(...)` при review cycles exhausted без gates
- **Поведение с gates:** Emergency gate prompt → можно skip/retry (работает корректно)
- **Поведение без gates:** Fatal error, весь run останавливается
- **Решение:** FinishTask("error", lastCommitSHA), записать статус в sprint-tasks.md, `continue` к следующей задаче. Откат не нужен.

#### BUG-3: Кеш-токены всегда 0 — неправильное имя JSON-поля

Ralph ищет в JSON поле `cache_read_tokens`, а Claude CLI отправляет `cache_read_input_tokens`. Из-за одного пропущенного слова `_input_` Ralph не видит кешированные токены.

- **Файл:** `session/result.go:38` — json-тег `"cache_read_tokens"`
- **CLI отправляет:** `"cache_read_input_tokens": 1447295`
- **Также пропущено:** `cache_creation_input_tokens` — нет поля вообще
- **Результат:** `cache_tokens: 0` во всём отчёте. Реальное использование — миллионы за run.
- **Решение:** Переименовать json-тег в `"cache_read_input_tokens"`, добавить поле `CacheCreationTokens` с тегом `"cache_creation_input_tokens"`

#### BUG-4: Модель всегда пустая строка — поле отсутствует в CLI output

Ralph ожидает поле `"model"` на верхнем уровне JSON-ответа, но Claude CLI его не отправляет. Модели указаны только внутри вложенного объекта `modelUsage`.

- **Файл:** `session/result.go:47` — парсит `msg.Model`
- **CLI отправляет:**
  ```json
  "modelUsage": {
    "claude-opus-4-6": {"inputTokens": 21, "outputTokens": 3551, ...},
    "claude-haiku-4-5-20251001": {"inputTokens": 3054, ...}
  }
  ```
- **Результат:** 20 WARN `unknown model pricing model=""`, fallback на `claude-opus-4-20250514`
- **Решение:**
  1. Добавить `model` в config.yaml (default: claude-opus-4-6)
  2. CLI флаг `--model` для override
  3. Fallback: самая мощная модель (самое дорогое ценообразование)
  4. Использовать `total_cost_usd` из CLI output напрямую — доверяем CLI

#### BUG-5: Стоимость занижена втрое — $20 вместо реальных ~$61

Следствие BUG-3 + BUG-4. Отдельного фикса не нужно — решается исправлением BUG-3 и BUG-4 + использованием `total_cost_usd` из CLI.

- **Ralph показал:** $20.24
- **Сумма CLI `total_cost_usd` из всех сессий:** ~$61
- **Расхождение:** 3x
- **Решение:** Следствие BUG-3 + BUG-4, отдельного фикса нет

### HIGH

#### BUG-6: Claude делает несколько задач за одну сессию

Claude видит весь sprint-tasks.md и в одной сессии делает 4-5 задач вместо одной. Коммиты сдвинуты на задачу — diff записывается не на ту задачу.

- **Пример:** В сессии BF-2.1 (730с, 76 turns) Claude сделал BF-2.1 + BF-2.2 + BF-2.3 + BF-2.4 + BF-3.1
- **Причина:** Промпт говорит "ONLY that task", но структурного ограничения нет
- **Решение (2 слоя):**
  1. **Поведенческий:** в execute.md секция "Session Completion" — закоммитил одну задачу → ОСТАНОВИСЬ
  2. **Правило 999:** запрет работать над задачами, которые не выбрал в начале сессии
  3. Полный sprint-tasks.md остаётся видимым — Claude сам выбирает задачу и видит контекст

#### BUG-7: input_tokens считает только raw input, не cache

- **В отчёте:** Total input_tokens = 6,422
- **Реальный input с cache:** миллионы токенов
- **Пример:** Сессия seq=2 показывает input_tokens=21, но cache_read_input_tokens=1,447,295
- **Решение:** 3 отдельных поля: `input_tokens` (raw) + `cache_read_tokens` + `cache_creation_tokens`. В summary показывать `total_input_tokens` как сумму.

#### BUG-8: error_max_turns не отражён в метриках ошибок

Две сессии (seq=14 BF-1.5, seq=16 BF-2.1) завершились с `"subtype": "error_max_turns"` и `"is_error": true`, но `exit_code=0`. Ralph записал их как успешные.

- **Файл:** Логика в `runner/runner.go` — проверяет только exit_code, не subtype
- **Результат:** Нет индикатора что работа была обрезана по лимиту turns
- **Решение:** Парсить `subtype` из JSON, при `is_error=true` и `exit_code=0` помечать сессию как `truncated`

### MEDIUM

#### BUG-9: Summary строка создаёт ложное впечатление провала

Вывод: `Run complete: 1 tasks (0 completed, 0 skipped, 1 failed)`. Выглядит как полная катастрофа, хотя 5 из 6 задач сделаны.

- **Причина:** Следствие BUG-1 (done vs completed)
- **Решение:** Следствие BUG-1, отдельного фикса нет

#### BUG-10: Количество findings не сходится

В итоге написано `Reviews: 3 cycles, 22 findings (1h/14m/6l)`, но 1+14+6 = 21, не 22. Одно из значений посчитано неверно.

- **Решение:** Проверить логику подсчёта в `runner/metrics.go`, исправить при реализации

#### BUG-11: Resume-сессия не считается в метриках

После 407 прокси-ошибки Ralph запустил resume-сессию (seq=1, $0.44, 62с). В директории 26 файлов, но JSON-отчёт показывает 25 sessions.

- **Решение:** Два файла на диске (для отладки), но одна логическая сессия в метриках — суммировать токены, стоимость, время с оригинальной сессией

#### BUG-12: Предупреждения `needs-formatting` без контекста

28 раз в логе `WARNING: Entry saved with [needs-formatting]` — но какие entries, почему нужно форматирование, что с этим делать — непонятно. Просто шум.

- **Решение:** Добавить контекст в warning: какой файл, какая entry

### ДИЗАЙН-ПРОБЛЕМЫ

#### DESIGN-1: Нет лимита на стоимость одной задачи — ОТКЛОНЕНО

BF-2.1 сожгла $12.14 (60% бюджета!) при среднем $1.62 на задачу.

- **Решение:** Отклонено. Лимита по стоимости не будет — хватает `max_review_iterations`.

#### DESIGN-2: Нет защиты от scope creep

Claude свободно правит файлы за пределами scope задачи.

- **Решение (3 точки):**
  1. В `execute.md` промпте — инструкция: трогать только файлы в scope текущей задачи
  2. В CLAUDE.md целевого проекта — аналогичное требование
  3. В одном из 5 ревью-агентов — строка: проверить что diff не выходит за scope

#### DESIGN-3: max_turns=75 мало для сложных frontend задач — ОТКЛОНЕНО

- **Решение:** Отклонено. Направление обратное — держать контекст маленьким (~100K токенов max). Если задача не влезла — продолжение в следующей итерации с чистым контекстом.

#### DESIGN-4: Ревью стоит столько же, сколько выполнение кода

41.5% стоимости прогона ушло на ревью.

- **Решение:** Определять режим ревью по diff size. Маленький diff → лёгкое ревью (Haiku), большой → полное (Sonnet). Порог определить при реализации.

#### DESIGN-5: Ложных срабатываний ревью ~75-80%

Три четверти findings — ложные: pre-existing проблемы, вне scope, неверная интерпретация.

- **Решение:** В промпт каждого из 5 ревью-агентов: критиковать ТОЛЬКО diff (изменения текущей задачи). Не критиковать pre-existing код. Контекст: вот задача X, вот изменения Y — оценивать только подход к решению X через Y.

#### DESIGN-6: Гидра-эффект — фикс одного бага рождает новый

BF-2.1: цикл 1→1, цикл 2→5, цикл 3→5, цикл 4→4, цикл 5→5.

- **Исследование:** Deep research (Nature 2025, DEV Community 2025) — проблема универсальна, ни один инструмент не решил полностью. Guardian Layer (детерминистические проверки) в 4x эффективнее prompt-only.
- **Решение (сейчас):** Трекать findings по циклам (1→5→5→4→5), показывать паттерн в отчёте, warning при гидре, но не останавливать — продолжать до max_review_iterations.
- **Решение (будущее, DESIGN-6b):** Ralph генерирует project-specific детерминистические проверки (lint, tests, hooks) как Guardian Layer перед LLM-ревью. Отдельный Epic.

### MINOR

#### MINOR-1: Нет прогрессии findings по циклам в итогах

Вместо `findings: 1→5→5→4→5` показан только агрегат `22 findings`.

- **Решение:** Показывать и агрегат, и прогрессию по циклам. Частично решается через DESIGN-6 (трекинг по циклам).

#### MINOR-2: Для проваленной задачи commit_sha пустой

BF-2.1 сделала 5 коммитов, но в JSON записано `commit_sha: ""`.

- **Решение:** При FinishTask("error") записывать последний известный HEAD SHA.

#### MINOR-3: Прокси-ошибка и ошибка Claude неразличимы

Оба дают exit_code=1. Неизвестно — сеть упала или Claude сломался.

- **Решение:** Парсить stderr/stdout на известные паттерны (407 Proxy, rate limit, connection refused), классифицировать ошибку в метриках.

#### MINOR-4: Timestamps не в UTC — ОТКЛОНЕНО

Время в JSON записано в local timezone (+03:00).

- **Решение:** Отклонено. Оставляем MSK (+03:00).

#### MINOR-5: Пути с обратными слэшами Windows

`"packages": ["apps\\backend\\src\\students"]` — на Linux не работают для фильтрации.

- **Решение:** `filepath.ToSlash()` — всегда прямые слэши в JSON. Работает и на Windows, и на Linux.

#### MINOR-6: Дубликаты findings между циклами ревью — ОТКЛОНЕНО

Одна и та же проблема описана дважды разными словами в разных циклах.

- **Решение:** Отклонено — это фича. Если проблема описана дважды — это информация о том, что ревьюер считает её важной или фикс не сработал. Ценно для анализа.

#### MINOR-7: Нет per-review-cycle времени в логах

Лог показывает время каждой сессии, но не общее время цикла.

- **Решение:** Добавить время полного цикла (execute + review + git + gate) в метрики.
