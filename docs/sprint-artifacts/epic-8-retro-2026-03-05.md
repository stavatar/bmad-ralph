## Challenges

### SerenaSyncFn Signature Change (Story 8.5)

Story 8.5 changed `SerenaSyncFn` return type from `error` to `(*session.SessionResult, error)` to propagate token/cost data for metrics. This required updating all 10 existing Story 8.4 test lambdas (`return nil` → `return nil, nil` / `return nil, err`). **Lesson:** When designing injectable functions, anticipate what data callers will need. Story 8.4 should have returned `SessionResult` from the start.

### Import Cycle in Story 8.4

Developer created local `syncTestGitClient` instead of using `testutil.MockGitClient` because `testutil` imports `runner` → cycle. This is a known constraint from Epic 3 (runner tests can't use testutil for runner-specific mocks). Not a new issue, but repeated.

### Validator Discoveries (6 fixes across 7 stories)

Validator agent found issues in 4 of 7 stories:
- **8.2:** `buildTemplateData` contradiction (story claimed Option B but AC could imply Option A)
- **8.4:** Trigger guard logic error (empty trigger should map to "task", not "run")
- **8.5:** `SessionMetrics` nil-safety not mentioned in AC
- **8.6:** Missing dependency on Story 8.5 (RecordSerenaSync accumulation)

Validator continues to add value — catching spec/code mismatches before development.

### HeadCommit Mock Slot Consumption

Per-task sync calls `HeadCommit` for `taskHeadBefore` capture. This consumed mock HeadCommit sequence slots in existing tests. Fixed by guarding the HeadCommit call with sync-enabled check (`SerenaSyncEnabled && trigger == "task"`). Pattern: mock-based tests are fragile to new call sites.

## Анализ: 30 findings при 0 новых паттернах

Ключевое наблюдение Epic 8: **findings и новые паттерны — разные метрики**. Findings — конкретные ошибки в коде конкретной story. Новые паттерны — ранее неизвестные *категории* ошибок, требующие добавления в `.claude/rules/`.

Все 30 findings Epic 8 попали в уже известные категории:

| Категория | Кол-во | Правило из Epic | Пример |
|-----------|--------|----------------|--------|
| Assertion quality (неполные проверки полей struct, отсутствие negative assertions) | ~12 | Epic 2 | M3 (8.5): тест проверяет только key+status, не все 5 nested полей |
| Doc comment accuracy (устаревший doc после рефакторинга) | 6 | Epic 3 | M2 (8.6): doc говорит "after execute loop", но per-task = during loop |
| Error path coverage (пропущен тест на error path) | 4 | Epic 4 | M2 (8.3): backupMemories — нет теста на inner error |
| Duplicate code (одинаковые тесты в разных файлах) | 3 | Epic 3 | M1 (8.6): PartialStatus = подмножество MultipleCalls |
| Code path coverage (untested branch) | 3 | Epic 4 | M3 (8.4): нет теста на empty trigger default |
| go fmt / formatting | 1 | Epic 3 | L2 (8.2): alignment после Edit tool |
| Test completeness (missing test case) | 1 | Epic 2 | L2 (8.1): beyond-range test case |

**Вывод:** Developer-агент *повторяет известные ошибки* в новом контексте. Правила описывают *что* проверять, но agent не всегда соблюдает их при генерации кода. Это проблема **enforcement**, а не **knowledge**.

**Возможные улучшения:**
1. **Developer prompt injection** — добавить в dev-story prompt топ-5 частых категорий findings как checklist перед завершением
2. **Reviewer prompt focus** — подсказывать reviewer'у конкретные категории для проверки
3. **PreToolUse hook расширение** — автоматическая проверка doc comments и assertion completeness

**Реалистичная оценка:** 4.3 findings/story — низкий уровень. Полный ноль вряд ли достижим: code review по определению находит то, что автор пропустил. Стабилизация на 3-5 findings/story при 0 HIGH/CRITICAL — это целевой уровень качества.

## Key Insights

1. **Zero new patterns = rules maturity peak** — 22→9→10→5→0 pattern decline across Epics 4-8. The `.claude/rules/` system has reached coverage saturation for this codebase
2. **30 findings при 0 новых паттернах = enforcement gap** — знание есть, применение неполное. Топ-3 повторяющихся категорий: assertion quality (40%), doc accuracy (20%), error path coverage (13%)
3. **Design pivots during planning are cheap** — Changing default trigger from "run" to "task" during sprint planning required editing 3 docs, 0 code. Same change post-implementation would have touched 5+ files
4. **Injectable function pattern is project standard** — SerenaSyncFn is the 4th injectable (after ReviewFn, DistillFn, GatePromptFn). Pattern is well-understood: nil = no capability, closure in Run(), mock in tests
5. **Per-task sync accumulation is non-trivial** — RecordSerenaSync changed from single-write to accumulator with derived status (success/partial/failed). Testing required 4 additional scenarios
6. **Agent team optimal at 28 tasks** — Fewer coordination issues than Epic 7 (40 tasks). Sweet spot: 7 stories x 4 pipeline steps
7. **Validator agent ROI confirmed** — 6 fixes across 7 stories (0.86/story). Catches spec mismatches that would become 2-3 review findings each
