# Анализ рисков: варианты эволюции ralph v2

## Дата: 2026-03-07

---

## Варианты

| ID | Название | Суть |
|----|----------|------|
| **A** | Bridge упрощается | Программный парсинг AC из story файлов, LLM убирается из bridge |
| **B** | Bridge убирается, runner со stories | Runner читает story файлы напрямую, sprint-tasks.md не нужен |
| **C** | Stories убираются, ralph с epics | Epic файл = единица работы, stories не создаются |
| **D** | ralph plan (самодостаточный) | Новая команда `ralph plan`: PRD/текст/issues -> sprint-tasks.md -> runner |
| **D+** | ralph plan + init + replan | Полный вариант D с инициализацией проекта и перепланированием |

### 1.1. Технические риски

| # | Риск | Вер. | Импакт | Score | Митигация |
|---|------|------|--------|-------|-----------|
| A-T1 | Регрессия качества декомпозиции: программный парсинг не понимает семантические связи между AC, группировка "1 AC = 1 task" грубее чем LLM | 2 | 3 | 6 | Runner всё равно читает source файл для деталей — грубая группировка компенсируется полным контекстом в runner. Task granularity rules остаются в format contract |
| A-T2 | Жёсткая привязка к структуре story: regex парсинг `## Acceptance Criteria` ломается при изменении BMad story формата | 3 | 3 | 9 | Defensive parsing с fallback: если structured AC не найдены, трактовать весь файл как plain text. Тесты на edge cases формата |
| A-T3 | Потеря keyword-based классификации: "already implemented", "manual" — список ключевых слов неполный, false positives | 2 | 2 | 4 | Начать с conservative списка (5-7 keywords). Добавлять по мере обнаружения ложных срабатываний. Config-опция для пользовательских keywords |
| A-T4 | Backward compatibility: sprint-tasks.md формат остаётся, но `source:` ссылки генерируются программно — возможны расхождения с ожиданиями runner | 1 | 2 | 2 | Format contract (sprint-tasks-format.md) не меняется. Runner парсит только `- [ ]` и `- [x]` — source поле информационное |

### 1.2. Продуктовые риски

| # | Риск | Вер. | Импакт | Score | Митигация |

## 8. Risk-adjusted рекомендация

### Ключевые выводы

1. **Вариант C (epics) — исключить.** Критический риск по всем категориям. Context window overflow + потеря human control = нефункциональный результат. Фактически, любая попытка сделать C работоспособным приводит к воссозданию stories (scope creep к B).

2. **Status quo — неприемлем долгосрочно.** Score 16.7 — второй по величине. Главные угрозы: adoption barrier (SQ-4, score 20), неработоспособность без BMad (SQ-7, score 25), bottleneck bridge (SQ-1, score 20). Все три не митигируются без архитектурных изменений.

3. **Вариант A — минимальный риск, минимальная ценность.** Score 4.1 — самый безопасный. Но решает только "испорченный телефон" (убирает LLM из bridge). НЕ решает зависимость от BMad (SQ-7), adoption barrier (SQ-4), time-to-first-task (SQ-1). Это оптимизация текущей архитектуры, не эволюция.

4. **Вариант D — оптимальный баланс риск/ценность.** Score 5.4 — умеренный риск. Решает ВСЕ ключевые проблемы status quo: adoption barrier (plain text input), BMad зависимость (опциональна), time-to-first-task (2-5 мин). Главный риск (D-R2, score 12) — качество plan.md промпта — митигируется итеративно.

5. **Вариант D+ — усиленный D с повышенным scope creep.** Score 8.5. Дополнительная ценность (init + replan) не пропорциональна дополнительному риску. Рекомендация: сначала D, потом D+ если D успешен.

6. **Вариант B — высокий риск при средней ценности.** Score 7.6. Главные проблемы: ломает scan.go + execute.md (B-T3, B-T5, score 12 каждый), объём недооценён (B-R1, score 12). Даёт ту же ценность что D (убирает bridge), но с большим количеством breaking changes.

### Рекомендуемая стратегия

**Фаза 1: Вариант D (ralph plan) — 4-5 stories, 2-3 дня**
- Новый пакет `planner/`, команда `ralph plan`
- Bridge остаётся как legacy
- Runner без изменений (sprint-tasks.md как контракт)
- Главный фокус: качество plan.md промпта (D-R2)

**Фаза 2: Deprecation bridge — 1 story, 0.5 дня**
- `ralph bridge` выдаёт warning, делегирует в `ralph plan`
- Документация обновляется

**Фаза 3: Удаление bridge — 1 story, 0.5 дня**
- Пакет bridge/ удаляется (-2844 строки)
- Нетто-эффект: -1324 строки (упрощение)

**Фаза 4 (опционально): D+ расширения — по необходимости**
- `ralph init` — если adoption требует scaffolding
- `ralph replan` — если пользователи запрашивают перепланирование
- Решение о Фазе 4 принимается после оценки adoption Фазы 3

### Что НЕ делать

- **НЕ делать C** — критический риск, нет реалистичной митигации
- **НЕ делать B напрямую** — D даёт ту же ценность с меньшим риском
- **НЕ делать D+ до успеха D** — scope creep увеличивает риск без доказанной ценности
- **НЕ оставлять status quo** — 5 из 7 рисков не митигируются без изменений
