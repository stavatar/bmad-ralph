# Верификация формулы приблизительного расчёта заполнения контекстного окна

Дата: 2026-03-06

## 1. Модель роста контекста

### Как реально растёт контекст в Claude Code

Согласно [официальной документации Anthropic](https://platform.claude.com/docs/en/build-with-claude/context-windows):

> "Progressive token accumulation: As the conversation advances through turns, each user message and assistant response accumulates within the context window. Previous turns are preserved completely."
> "Linear growth pattern: The context usage grows linearly with each turn, with previous turns preserved completely."

Таким образом, **Anthropic сама описывает рост как линейный**. Однако на практике рост **ступенчатый**:

- **Фиксированный overhead на каждом turn'е**: system prompt (~2.7K токенов) + tool definitions (~16.8K токенов) — это ~20K токенов, которые присутствуют в каждом API-запросе
- **Переменные скачки**: чтение файлов (tool results) может добавить тысячи токенов за один turn
- **Output accumulation**: каждый assistant response становится частью входных данных на следующем turn'е

В среднем, при достаточном числе turn'ов, ступенчатый рост приближается к линейной модели. Для сессий с 20+ turn'ами линейная аппроксимация разумна.

### Формула линейного роста

Если контекст растёт линейно от `C_0` (начальный overhead) до `C_max` за N turn'ов:
- Turn i видит примерно `C_0 + i × (C_max - C_0) / N` входных токенов
- Кумулятивная сумма ≈ `N × C_0 + N × (C_max - C_0) / 2` = `N × (C_0 + C_max) / 2`

## 2. Анализ компонентов

### Что входит в context window на каждом turn'е

Согласно [документации по prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching) и [контекстным окнам](https://platform.claude.com/docs/en/build-with-claude/context-windows):

**Все три типа input-токенов занимают место в context window:**

```
context_fill_per_turn = input_tokens + cache_creation_input_tokens + cache_read_input_tokens
```

- **`cache_read_input_tokens`** — токены прочитанные из кеша (system prompt, tools, CLAUDE.md, ранняя часть conversation history). Стоят 0.1x, но **занимают то же место** в окне
- **`cache_creation_input_tokens`** — новые токены записанные в кеш на этом turn'е (новые части conversation, tool results). Стоят 1.25x
- **`input_tokens`** — токены после последнего cache breakpoint'а, не подлежащие кешированию

**Output-токены НЕ входят в `used_percentage` контекста на текущем turn'е**, но они становятся input-токенами на следующем turn'е (как часть conversation history).

Это подтверждается [codelynx.dev](https://codelynx.dev/posts/calculate-claude-code-context):
> "Output tokens generated by Claude are not included in the context calculation — only input-related tokens count toward your usage percentage."

И [GitHub issue #66](https://github.com/anthropics/claude-agent-sdk-typescript/issues/66) с формулой:
```
totalContext = input_tokens + cache_creation_input_tokens + cache_read_input_tokens
percentage = (totalContext / contextWindowSize) * 100
```

### Ключевое разграничение: per-turn vs cumulative

Это **критически важный** момент, обнаруженный в ходе исследования:

| Источник данных | Содержит | Использование |
|---|---|---|
| `result.usage` (из финального JSON) | **Кумулятивные** суммы за всю сессию | Расчёт стоимости |
| Последнее `message.usage` (per-turn) | Токены **одного** API-запроса | Context fill |

**Ralph получает только кумулятивные данные** из финального result JSON. Per-turn данные доступны через statusline JSON (`context_window.current_usage`), но не в result JSON.

## 4. Проблемы базовой формулы

### 4.1. Игнорирование cache_creation и input_tokens

Формула использует только `cache_read`. Но на каждом turn'е context fill = `cache_read + cache_creation + input_tokens`:

- **`cache_creation`**: на первых turn'ах CLAUDE.md и system prompt записываются в кеш. На поздних turn'ах новые tool results и conversation history записываются в кеш. Это реальное место в контексте.
- **`input_tokens`**: токены после последнего cache breakpoint'а (обычно маленькая часть, но не нулевая).

Для сессии #1: `cache_creation = 56,760`, `input = 25` — cache_creation составляет ~2.5% от cache_read. Это не пренебрежимо малая величина.

### 4.2. Неточность линейной модели

Линейная модель предполагает `context[i] = i × C_max / N`. На практике:
- **Turn 1** уже имеет ~20K токенов (system prompt + tools) — не 0
- **Рост неравномерный**: чтение большого файла на turn 5 даёт скачок, а run команды на turn 10 — маленький прирост
- При большом начальном overhead формула **занижает** максимальное заполнение

### 4.3. Output-токены как "скрытый" вклад

Output на turn i становится input на turn i+1 (как cached read). Таким образом output **косвенно** учтён в cache_read следующих turn'ов. Это не ошибка формулы — двойного счёта нет.

### 4.4. Поведение при близости к пределу

Если сессия приближается к auto-compact threshold (~83%), формула должна давать ~83%. Для данных это не проверяется (нет сессий с компактом), но теоретически формула корректна в этом отношении.

### 4.5. Начальный overhead не учтён

System prompt (~2.7K) + tools (~16.8K) = ~20K токенов присутствуют в **каждом** turn'е. Это фиксированная часть, не растущая линейно. Кумулятивный вклад = `N × 20K`, а линейная модель это "размазывает" по росту от 0 до C_max, что завышает C_max.

## 5. Улучшенная формула

### Подход 1: Полная формула с учётом всех компонентов

Кумулятивные данные за сессию содержат суммы per-turn значений:
```
cumulative_total = SUM(per_turn_input + per_turn_cache_read + per_turn_cache_creation)
```

Для линейной модели с начальным overhead C_0:
```
cumulative_total = N × C_0 + N × (C_max - C_0) / 2
```

Откуда:
```
C_max = 2 × (cumulative_total / N) - C_0
```

Где:
- `cumulative_total = cache_read + cache_creation + input_tokens`
- `C_0 ≈ 20,000` (system prompt + tools, можно настроить)
- `N = num_turns`

```
approx_max_fill_pct = C_max / context_window × 100
                    = (2 × (cache_read + cache_creation + input) / N - C_0) / context_window × 100
```

### Расчёт по улучшенной формуле (C_0 = 20,000)

### Подход 2: Упрощённая формула без C_0

Если C_0 неизвестен (Ralph не знает размер system prompt):

```
approx_max_fill_pct = 2 × (cache_read + cache_creation + input) / (num_turns × context_window) × 100
```

Это даёт результаты на ~3-5% выше, чем подход 1, что является верхней оценкой (upper bound).

### Подход 3: Прямая оценка через отношение к max_context

Для наиболее длинных сессий (num_turns > 30) можно использовать эмпирическую формулу:

```
approx_max_fill_pct ≈ (cache_read + cache_creation + input) / (num_turns × context_window / 2) × 100
```

Что математически эквивалентно подходу 2 — это та же формула, записанная иначе.

## 6. Поведение при компакте

### Что происходит с кумулятивными данными при auto-compact

Согласно [документации по compaction](https://platform.claude.com/docs/en/build-with-claude/compaction) и [исследованию context buffer](https://claudefa.st/blog/guide/mechanics/context-buffer-management):

1. **Auto-compact срабатывает** при ~83.5% заполнения (167K из 200K)
2. **Conversation history сжимается** до summary
3. **Кумулятивные токены продолжают расти** — compaction не сбрасывает cumulative usage в result JSON
4. **Per-turn usage (current_usage) резко падает** после compaction — следующий turn видит сжатый контекст

### Влияние на формулу

**Компакт ломает линейную модель**:

- До компакта: контекст рос до 167K
- После компакта: контекст упал до ~30-40K (summary)
- Затем снова растёт линейно

Кумулятивный cache_read будет содержать:
1. Сумму линейно растущих turn'ов ДО компакта
2. Резкое падение
3. Сумму линейно растущих turn'ов ПОСЛЕ компакта

Формула `2 × cache_read / N` даст **заниженное** значение, т.к. среднее уменьшится из-за пост-компактных turn'ов с маленьким контекстом.

### Детекция компакта

Ralph может детектировать компакт по:
- Непропорционально большому num_turns для данного cache_read
- Отношению `cache_read / num_turns²` значительно меньше ожидаемого
- Наличию `total_cost_usd` выше ожидаемого для данного fill_pct (токены до компакта были потрачены, но "стёрты")

**Важно**: при наличии компакта формула даёт максимальное заполнение **после последнего компакта**, а не абсолютный максимум за всю сессию. Без per-turn данных точнее оценить невозможно.

## 7. Рекомендация

### Финальная формула для Ralph

**Рекомендуется подход 2** (упрощённый, без C_0):

```go
func ApproxContextFillPct(m *SessionMetrics, contextWindow int) float64 {
    if m.NumTurns == 0 || contextWindow == 0 {
        return 0
    }
    totalCumulative := float64(m.CacheReadTokens + m.CacheCreationTokens + m.InputTokens)
    estimatedMax := 2.0 * totalCumulative / float64(m.NumTurns)
    return estimatedMax / float64(contextWindow) * 100.0
}
```

### Обоснование выбора

1. **Использует все доступные компоненты** — cache_read + cache_creation + input (в базовой формуле был только cache_read)
2. **Не требует знания C_0** — system prompt/tools overhead варьируется между проектами
3. **Даёт верхнюю оценку** — завышает на ~10% относительно точной формулы с C_0, что безопаснее для мониторинга (лучше предупредить раньше)
4. **Работает с имеющимися данными** — только кумулятивные значения из result JSON

### Оценка погрешности

- **Для сессий без компакта (< 83% fill)**: погрешность ±10-15%
  - Завышение из-за отсутствия C_0 коррекции: ~+10%
  - Неравномерность роста: ±5%
- **Для сессий с компактом**: формула **занижает** реальный максимум. В этих случаях реальный пик был ~83%, а формула покажет меньше
- **Для очень коротких сессий (< 10 turn'ов)**: погрешность выше из-за малой выборки

### Сравнение с базовой формулой

| Аспект | Базовая (только cache_read) | Улучшенная (all input) |
|--------|----------------------------|----------------------|
| Компоненты | cache_read | cache_read + cache_creation + input |
| Погрешность | ±15-20% | ±10-15% |
| Тенденция | Занижает на 2-4% (пропускает creation+input) | Завышает на ~10% (нет C_0) |
| Компакт | Не обнаруживает | Не обнаруживает |
| Рекомендация | Приемлемо для грубой оценки | Лучше для мониторинга |

### Оптимальная стратегия для Ralph

Для более точного мониторинга рекомендуется также:

1. **Парсить per-turn данные из statusline JSON или transcript**, если они доступны — это даст точный `used_percentage` без аппроксимаций
2. **Добавить флаг `compacted bool`** в результаты (если Claude Code его предоставляет), чтобы маркировать сессии с компактом
3. **Использовать формулу как fallback** — только когда per-turn данные недоступны
