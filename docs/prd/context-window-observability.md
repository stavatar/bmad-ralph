## Контекст и обоснование

Исследования 2024-2026 годов доказывают: деградация качества LLM при заполнении контекстного окна — измеримый факт ("context rot", Chroma Research 2025). Три зоны качества:

- **0-50% fill** (зелёная зона) — максимальное качество, retrieval ~95%, reasoning без деградации
- **50-75% fill** (жёлтая зона) — "lost in the middle", retrieval ~80-90%, recency bias
- **75-100% fill** (красная зона) — compaction, галлюцинации, потеря инструкций

Анализ реальных данных mentorlearnplatform (11 execute-сессий, `max_turns: 50`) подтверждает:

| Сессии по turns | Approx context fill | Зона |
|-----------------|---------------------|------|
| 18-22 turns | 45-65% | Жёлтая |
| 25-35 turns | 58-80% | Жёлтая-красная |
| 38-41 turns | 69-88% | Жёлтая-красная |

При `max_turns: 50` все 11 сессий оказались в жёлтой или красной зоне (52-88% fill). При `max_turns: 15` расчётный fill ~30-35% — глубоко в зелёной зоне.

**Ключевое решение:** `max_turns` — главный защитный механизм. Ralph поддерживает resume — задача, не завершённая за N turns, продолжается в следующей итерации со свежим контекстом. Метрики — для подтверждения и мониторинга, а не для runtime-решений.

## Отвергнутая альтернатива: stream-json

Исследование [compaction-metrics.md](../research/compaction-metrics.md) рекомендовало `--output-format stream-json` как наиболее точный подход — парсинг NDJSON потока с `CompactBoundaryMessage` маркерами и per-turn `usage` данными.

**Причины отказа:**
- **Blast radius:** замена `--output-format json` на `stream-json` требует переработки `session.Execute()` (buffer → streaming parser), `session.ParseResult()` (JSON → NDJSON), `session.Options`, и всех mock binary сценариев — ~290 LOC production + ~300 LOC тестов
- **Маргинальный выигрыш:** формула fill% с погрешностью ±10-15% достаточна для мониторинга. PreCompact hook даёт точный compaction count без изменения session layer
- **Непроверенность:** `CompactBoundaryMessage` документирован для Agent SDK, но не верифицирован в реальном CLI output `claude -p --output-format stream-json`

Выбранный подход (hook + формула) даёт **~490 LOC** вместо **~590 LOC**, с нулевыми изменениями session layer и mock binary.

## Scope

### Входит

- Снижение дефолтного `max_turns` с 50 до 15
- PreCompact hook + temp-файл счётчик для подсчёта compactions
- Функция `EstimateMaxContextFill` из кумулятивных token данных (уже доступных в `SessionMetrics`)
- Новые поля в `TaskMetrics` / `RunMetrics` для context fill и compactions
- Warning система с настраиваемыми порогами
- Строка Context в run summary
- Расширение session log header

### Не входит

- Переход на `--output-format stream-json` (см. секцию выше)
- Изменение `session.Execute()`, `session.ParseResult()`, `session.Options`
- Изменение mock binary
- Изменение bridge
- Real-time kill сессии при превышении порога — Growth
- Adaptive max_turns — Growth
- Per-turn данные — не нужны при правильном max_turns

## Функциональные требования

### A. Снижение max_turns (главный механизм)

- **FR81:** Дефолтное значение `max_turns` в `config/defaults.yaml` снижено с 50 до 15. Обоснование: при 15 turns приблизительный context fill ~30-35%, зелёная зона (0-50%). Задачи, не завершённые за 15 turns, продолжаются в следующей итерации через resume со свежим контекстом. Пользователь может вернуть 50 через config или CLI флаг `--max-turns`

### B. Compaction count через PreCompact hook

- **FR82:** Ralph перед каждым вызовом `session.Execute()` создаёт temp-файл (через `os.CreateTemp`) для подсчёта compactions. Путь к файлу передаётся Claude Code через environment variable `RALPH_COMPACT_COUNTER` (добавляется в `session.Options.Env`). Temp-файл создаётся пустым (0 байт). После завершения сессии файл удаляется (defer cleanup)

- **FR83:** Ralph при запуске (`ralph run`) обеспечивает наличие PreCompact hook:

  **Hook-скрипт:** `.ralph/hooks/count-compact.sh`
  ```bash
  #!/bin/bash
  # Auto-generated by Ralph. Counts auto-compact events.
  [ -n "$RALPH_COMPACT_COUNTER" ] && echo 1 >> "$RALPH_COMPACT_COUNTER"
  ```

  **Регистрация в `.claude/settings.json`:**
  ```json
  {
    "hooks": {
      "PreCompact": [
        {
          "matcher": "auto",
          "hooks": [
            {
              "type": "command",
              "command": ".ralph/hooks/count-compact.sh"
            }
          ]
        }
      ]
    }
  }
  ```

  **Логика обеспечения (ensure, не overwrite):**
  1. Если `.ralph/hooks/count-compact.sh` не существует — создать, сделать `chmod +x`
  2. Если существует но содержимое не совпадает с текущей версией — перезаписать (обновление Ralph)
  3. Если `.claude/settings.json` не существует — создать с hook конфигурацией
  4. Если существует — прочитать JSON, проверить наличие `hooks.PreCompact` записи с `matcher: "auto"` и command `.ralph/hooks/count-compact.sh`. Если записи нет — добавить в массив, сохранить. Если есть — не трогать. **Никогда не удалять** другие записи пользователя в settings.json
  5. При ошибке записи settings.json — warning в лог, продолжить без hook (graceful degradation)

- **FR84:** После завершения `session.Execute()` Ralph читает файл-счётчик. Количество непустых строк = количество compactions. Значение передаётся в `MetricsCollector.RecordSession()` как дополнительный параметр. Temp-файл удаляется после чтения. При ошибке чтения (файл не найден, permission) — compactions = 0, warning в лог (graceful degradation)

### C. Приблизительный context fill % (формула из кумулятивных данных)

- **FR85:** Новая функция в пакете `runner`:
  ```go
  func EstimateMaxContextFill(metrics *session.SessionMetrics, contextWindow int) float64
  ```
  Формула (верифицирована на 11 реальных сессиях, погрешность ±10-15%):
  ```
  total_cumulative = cache_read + cache_creation + input_tokens
  effective_turns = max(num_turns, 2)
  estimated_max = 2 × total_cumulative / effective_turns
  fill_pct = estimated_max / context_window × 100
  ```

  Guard `max(num_turns, 2)`: при `numTurns == 1` формула даёт `2 × total`, что равно двойному переcчёту (один turn = всё в первом вызове, нет "роста"). Minimum 2 предотвращает завышение.

  При `numTurns == 0` или `contextWindow == 0` возвращает 0.0.

  Формула даёт верхнюю оценку (upper bound), что безопаснее для мониторинга — лучше предупредить раньше.

  **Числовой пример** (реальная сессия #4 из mentorlearnplatform):
  ```
  cache_read = 1,456,521    cache_creation = 57,388    input = 2,700
  num_turns = 25            context_window = 200,000

  total_cumulative = 1,456,521 + 57,388 + 2,700 = 1,516,609
  effective_turns = max(25, 2) = 25
  estimated_max = 2 × 1,516,609 / 25 = 121,329
  fill_pct = 121,329 / 200,000 × 100 = 60.7%
  ```
  Сессия с 25 turns показывает ~61% fill — жёлтая зона, что согласуется с ожидаемым для `max_turns: 50`.

  **Edge case** (1 turn):
  ```
  cache_read = 20,000    cache_creation = 5,000    input = 500
  num_turns = 1          context_window = 200,000

  total_cumulative = 25,500
  effective_turns = max(1, 2) = 2
  estimated_max = 2 × 25,500 / 2 = 25,500
  fill_pct = 25,500 / 200,000 × 100 = 12.8%
  ```
  Без guard: `2 × 25,500 / 1 = 51,000` → 25.5% — двойное переcчётание.

- **FR86:** `context_window` берётся из `SessionMetrics.ContextWindow` (уже парсится из `modelUsage` в result JSON). Fallback: 200000 (Claude Code default)

### D. Метрики и warning

- **FR87:** Новые поля в `TaskMetrics`:
  - `TotalCompactions int` — сумма compactions по всем сессиям задачи
  - `MaxContextFillPct float64` — максимальный % fill среди сессий задачи

- **FR88:** Новые поля в `RunMetrics`:
  - `TotalCompactions int` — общее количество compactions за весь ран
  - `MaxContextFillPct float64` — максимальный % fill за весь ран

- **FR89:** Warning в `RunLogger` после каждой сессии:
  - Тихо (только в session log) при fill ≤ warn порога
  - WARN при fill > warn порога и ≤ critical порога: `"context fill NN.N%% — consider reducing max_turns (current: N) or splitting task into smaller pieces"`
  - ERROR при fill > critical порога: `"context fill NN.N%% exceeds critical threshold — quality degradation likely, reduce max_turns (current: N)"`
  - ERROR при compactions > 0: `"N compaction(s) detected — context was compressed, quality degraded. Reduce max_turns (current: N)"`

  Каждый warning содержит **actionable guidance**: что сделать (reduce max_turns) и текущее значение для контекста.

- **FR90:** Строка Context в `formatSummary` (`cmd/ralph/run.go`):
  ```
  Context: max NN.N% fill, N compactions
  ```
  При compactions > 0 — строка дополняется `[!]` маркером и окрашивается жёлтым (fatih/color). При fill > critical порога — окрашивается красным

### E. Config

- **FR91:** Два новых поля в `config.Config`:
  - `ContextWarnPct int` (yaml: `context_warn_pct`, default: 55) — порог для WARN
  - `ContextCriticalPct int` (yaml: `context_critical_pct`, default: 65) — порог для ERROR
  Валидация: оба значения в диапазоне 1-99, `context_critical_pct > context_warn_pct`. Дефолты добавляются в `config/defaults.yaml`

  **Обоснование порогов:** 55% — начало жёлтой зоны деградации (50%+), с учётом погрешности формулы +10% реальный fill ~45-50%. 65% — глубокая жёлтая зона, реальный fill ~55-60%, качество retrieval уже ~80-90%

### F. Session log расширение

- **FR92:** Header session log файла расширяется:
  ```
  === SESSION execute seq=3 exit_code=0 elapsed=45.2s compactions=0 max_fill=42.7% ===
  ```
  Поля `compactions` и `max_fill` добавляются в `SessionLogInfo`

## Нефункциональные требования

- **NFR30:** PreCompact hook — best effort. При любом сбое (файл не создан, hook не сработал, permission denied) — compactions = 0 с warning в лог. Основной pipeline не блокируется

- **NFR31:** Формула `EstimateMaxContextFill` работает только с кумулятивными данными из существующего result JSON. Не требует изменения session layer, mock binary или output формата

- **NFR32:** Новые config поля имеют sensible defaults (`context_warn_pct: 55`, `context_critical_pct: 65`). `ralph run` без изменений config работает с предупреждениями на дефолтных порогах

- **NFR33:** Снижение `max_turns` до 15 не ломает существующие workflow. Пользователь может вернуть любое значение через config или CLI флаг `--max-turns`

- **NFR34:** Temp-файл счётчик compactions создаётся в `os.TempDir()` с уникальным именем (паттерн `ralph-compact-*`), удаляется после чтения. При crash — OS cleanup через стандартный механизм temp-файлов

- **NFR35:** Hook-скрипт `count-compact.sh` генерируется Ralph'ом, не требует ручного создания. При обновлении Ralph обновляет скрипт если содержимое изменилось. settings.json обновляется аддитивно (merge), никогда не перезаписывается полностью

## User Stories (высокоуровневые)

### US9: Защита от context rot через снижение max_turns
**Как** пользователь Ralph, **я хочу** чтобы дефолтный `max_turns` обеспечивал работу в зелёной зоне контекста (≤50% fill), **чтобы** получать максимальное качество агентных сессий без ручной настройки.
- **AC1:** Дефолтный `max_turns` = 15, расчётный context fill ~30-35%
- **AC2:** Задачи, не завершённые за 15 turns, продолжаются через resume
- **AC3:** Пользователь может увеличить `max_turns` через config/CLI

### US10: Мониторинг context fill и compactions
**Как** пользователь Ralph, **я хочу** видеть context fill % и количество compactions после каждого рана, **чтобы** принимать обоснованные решения о настройке max_turns и размере задач.
- **AC1:** Run summary содержит строку `Context: max NN.N% fill, N compactions`
- **AC2:** WARN/ERROR в лог при превышении порогов (55%/65%, настраиваемые)
- **AC3:** ERROR при обнаружении compaction с рекомендацией уменьшить max_turns
- **AC4:** Session log содержит compactions и max_fill в header

## Риски и митигации

| Риск | Вероятность | Impact | Митигация |
|------|-------------|--------|-----------|
| Формула context fill неточна (±10-15%) | Высокая | Low | Верифицирована на 11 реальных сессиях. Даёт upper bound — безопаснее для мониторинга. При max_turns:15 погрешность не влияет на решения (fill ~30% далеко от порогов) |
| PreCompact hook не срабатывает (Claude Code bug) | Низкая | Low | Graceful degradation: compactions = 0. Формула fill покажет > 83% если компакт был нужен |
| max_turns: 15 недостаточно для сложных задач | Средняя | Medium | Пользователь настраивает через config/CLI. Resume продолжает задачу со свежим контекстом. Для тяжёлых проектов рекомендуется 20-25 |
| RALPH_COMPACT_COUNTER env var конфликтует с Claude Code | Очень низкая | Medium | Уникальный префикс `RALPH_`. Claude Code не использует env vars с этим префиксом |
| Hook settings.json конфликтует с пользовательскими hook'ами | Средняя | Medium | Аддитивный merge: Ralph добавляет свою запись в массив PreCompact, никогда не удаляет чужие записи |
| False positive warnings при пороге 55% | Низкая | Low | Пороги настраиваемые через config. При max_turns:15 fill ~30-35%, далеко от порога 55% |
| Формула занижает fill при наличии compaction | Средняя | Medium | PreCompact hook детектирует сам факт compaction — warning выдаётся по compaction count, не только по формуле |
| numTurns=1 edge case | Низкая | Low | Guard `max(numTurns, 2)` предотвращает 2x завышение |
