## Контекст

При тестировании Ralph Run на реальном проекте (mentorlearnplatform, 15 задач, 3 рана) выявлены 4 системные проблемы, приводящие к напрасной трате токенов и некорректному поведению:

1. **Повторное выполнение задач** — Ralph тратит $1-2 на задачу, которая уже закоммичена, потому что sprint-tasks.md рассинхронизирован с git history
2. **WSL/Windows path баг** — дистилляция падает из-за несовместимости путей между WSL и Windows Go
3. **Гидра-паттерн** — review цикл не сходится: замечания 3→2→4→1→0, стоимость одной задачи $20+ при 5 циклах
4. **Scope creep** — Claude реализует соседнюю задачу из sprint-tasks.md вместо текущей

### Метрики проблем

| Проблема | Потери (наблюдённые) | Частота | Impact |
|----------|---------------------|---------|--------|
| Повторное выполнение | $1.70-$3.50 на ран | При перегенерации sprint-tasks.md | Средний |
| Path баг | Distillation failure | На каждом ране в WSL | Высокий |
| Гидра | $20.29 на одну задачу (BF-3.3) | ~20% задач с [GATE] | Критический |
| Scope creep | 1 задача выполнена вместо другой | Наблюдалось 1 раз из 15 задач | Средний |

## Scope

### Входит

- Pre-flight проверка задач перед execute
- Маркер задачи в commit message для трассировки
- Smart merge сохранение `[x]` при перегенерации
- Кросс-платформенная нормализация путей (filepath.Join/Abs)
- Graceful degradation для некритических файловых операций
- Прогрессивная схема review (severity порог + scope lock + бюджет замечаний)
- Эскалация модели и effort level на поздних циклах
- Статистика по review sub-agents
- Усиление scope boundary в execute промпте
- Проверка scope creep в implementation review agent

### Не входит

- Детерминированные pre-review gates (линтеры, go vet) — отдельный epic
- Risk scoring задач при декомпозиции — отдельный epic
- Fresh context reset для упорных случаев гидры — Growth
- Кумулятивный файл статистики по агентам — делается вручную поверх JSON-логов

## Функциональные требования

### Pre-flight и трассировка задач

- **FR67:** Execute-промпт требует от Claude включать маркер `[task:<хэш>]` в конце commit message. Хэш — первые 6 символов SHA-256 от текста задачи (строка после `] ` в sprint-tasks.md). Формат: `[task:a1b2c3]`. Ralph вычисляет хэш из описания задачи и использует для матчинга с git log

- **FR68:** Перед запуском execute для задачи `[ ]` система выполняет pre-flight проверку:
  1. Вычисляет хэш текста задачи
  2. Ищет `[task:<хэш>]` в `git log --oneline -20`
  3. Если коммит найден И `review-findings.md` пуст/отсутствует → задача уже выполнена, помечается `[x]`, execute пропускается
  4. Если коммит найден И `review-findings.md` содержит findings → стандартный цикл execute (исправление замечаний)
  5. Если коммит не найден → стандартный цикл execute
  Pre-flight результат логируется: `INFO pre-flight skip` или `INFO pre-flight: proceed`

- **FR69:** При перегенерации sprint-tasks.md через bridge (smart merge) система сохраняет пометки `[x]` из предыдущей версии файла. Матчинг задач — по хэшу текста (тот же алгоритм что в FR67). Задачи с `[x]` в старой версии, чей хэш совпадает с задачей в новой версии, переносят статус `[x]`

### Кросс-платформенные пути

- **FR70:** Все конструирования путей к файлам в runner используют `filepath.Join` и `filepath.Abs` из Go stdlib вместо строковой конкатенации. Это обеспечивает корректную работу на Windows, WSL и Linux без ручной конвертации путей

- **FR71:** Некритические файловые операции (дистилляция LEARNINGS.md, запись knowledge) при ошибке `os.IsNotExist` выполняют graceful skip с `WARN` в лог вместо abort рана. Дистилляция, запись learnings и аналогичные операции — best-effort, их отказ не блокирует цикл execute→review

### Прогрессивная схема review (анти-гидра)

- **FR72:** Дефолтное значение `max_review_iterations` изменяется с 3 на 6. Прогрессивная схема применяет три механизма, параметры которых зависят от номера цикла:

  | Цикл | Severity порог | Scope diff | Макс. замечаний | Модель execute |
  |------|---------------|------------|-----------------|----------------|
  | 1 | LOW+ | Полный | 5 | Стандартная |
  | 2 | LOW+ | Полный | 5 | Стандартная |
  | 3 | MEDIUM+ | Инкрементальный | 3 | Макс. + effort=high |
  | 4 | HIGH+ | Инкрементальный | 1 | Макс. + effort=high |
  | 5 | CRITICAL | Инкрементальный | 1 | Макс. + effort=high |
  | 6 | CRITICAL | Инкрементальный | 1 | Макс. + effort=high |

  Значения в таблице — дефолты. Пользователь может переопределить `max_review_iterations` через конфиг или CLI, при этом прогрессия масштабируется пропорционально

- **FR73:** Severity filtering в `DetermineReviewOutcome`: findings с severity ниже текущего порога цикла записываются в лог (`INFO finding below threshold`), но не блокируют pipeline и не попадают в review-findings.md. Реализует FR16a с динамическим порогом вместо статического

- **FR74:** Scope lock начиная с цикла 3: review получает инкрементальный diff (`git diff HEAD~1..HEAD`) вместо полного diff задачи. В контекст review передаются:
  - Описание задачи и ссылка на story (для понимания цели)
  - Findings предыдущего цикла (для контекста исправлений)
  - Инструкция: "проверь корректность исправлений и отсутствие новых проблем уровня <порог>+"

- **FR75:** Бюджет замечаний: review-промпт ограничивает количество confirmed findings за один цикл. Значение бюджета определяется номером цикла (см. таблицу FR72). Review-промпт содержит инструкцию: "Найди НЕ БОЛЕЕ N самых важных замечаний. Приоритизируй по severity"

- **FR76:** Начиная с цикла 3, execute-сессия запускается с переменной окружения `CLAUDE_CODE_EFFORT_LEVEL=high`. Это активирует extended thinking (глубокое рассуждение) для более качественного исправления сложных замечаний на поздних циклах

### Статистика по review агентам

- **FR77:** Формат findings в review-findings.md расширяется 5-м полем `Агент`:
  ```
  ### [SEVERITY] Finding title
  - **Описание**: ...
  - **Файл**: ...
  - **Строка**: ...
  - **Агент**: quality|implementation|simplification|design-principles|test-coverage
  ```
  Review-промпт и промпты sub-agents обновляются для включения имени агента в каждый finding

- **FR78:** Ralph при парсинге review-findings.md извлекает поле `Агент` и severity каждого finding. Агрегированная статистика записывается в JSON-лог рана (`.ralph/logs/ralph-run-<id>.json`) в секцию `agent_stats`:
  ```json
  "agent_stats": {
    "<agent_name>": {"critical": N, "high": N, "medium": N, "low": N}
  }
  ```
  Статистика аккумулируется за весь ран (все задачи, все циклы). При отсутствии поля `Агент` в finding — finding учитывается в категории `unknown`

### Защита от scope creep

- **FR79:** Execute-промпт (`execute.md`) содержит блок `SCOPE BOUNDARY (MANDATORY)`:
  - Явный запрет реализации задач, отличных от текущей `__TASK__`
  - Инструкция проверить перед коммитом: все изменённые файлы и строки относятся к текущей задаче
  - При обнаружении изменений для другой задачи — откат через `git checkout`
  - Если задача зависит от другой незавершённой — остановиться и сообщить, не реализовывать обе

- **FR80:** Implementation review agent (`agents/implementation.md`) проверяет scope compliance:
  - Все изменения в diff относятся к AC текущей задачи
  - Если обнаружены изменения, реализующие другую задачу из sprint-tasks.md — это finding severity HIGH
  - Формат: "Scope creep: изменения в <файл> реализуют задачу '<описание другой задачи>', а не текущую '<описание текущей задачи>'"

## Риски и митигации

| Риск | Вероятность | Impact | Митигация |
|------|-------------|--------|-----------|
| Claude игнорирует маркер `[task:хэш]` в commit message | Средняя | Low | Pre-flight = best-effort, при отсутствии маркера — стандартный цикл |
| Хэш коллизия (два разных текста → один хэш) | Очень низкая | Low | 6 символов SHA-256 = 16M комбинаций, достаточно для sprint-tasks |
| Прогрессивная схема пропускает реальные LOW findings | Средняя | Medium | LOW findings логируются всегда, доступны для ручного review |
| Claude игнорирует scope boundary инструкцию | Низкая | Medium | Двойная защита: промпт execute + проверка в review agent |
| Инкрементальный diff на цикле 3+ пропускает regression | Низкая | High | Review получает предыдущие findings как контекст + полное описание задачи |
| EFFORT_LEVEL=high значительно увеличивает стоимость | Высокая | Medium | Применяется только с цикла 3, до этого стандартная модель |
